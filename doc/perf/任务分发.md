# 性能优化任务分发文档

> 目标：在保证 59/59 功能正确性的前提下，提升各基准相对 `gcc -O2` 的指令数 ratio。
> Baseline 见 `doc/perf/baseline.md`。当前平均 ratio ≈ 0.69（指令数口径）。

## 度量口径（统一）

- **核心指标**：动态指令总数（`spike -g` histogram 求和），平台无关、零噪声、可复现。
- 评测脚本：`SPIKE=/usr/local/bin/spike bash tests/bench_icount.sh`
- ratio = gcc指令数 / 我方指令数，**越高越好**，1.0 = 持平 gcc -O2。
- **硬约束**：每个改动必须通过
  `SPIKE=/usr/local/bin/spike TIMEOUT="timeout 20" bash tests/run_tests.sh` 与 `-opt`
  两个 59/59，且 `run_opt_compare.sh` 为 0 OPT_BUG。功能错误的优化一律回滚。

## 重要前提：IR 非 SSA

局部变量跨赋值复用同一 vreg（`r=r+1` 重定义同一 vreg）。任何值传播/CSE/LICM 类 pass
**必须块内带 kill 或做严格到达定义分析**，否则越过重定义点会破坏语义（已踩坑两次，见
`src/opt/Passes.cpp` 中 copyProp/licm 的注释）。要做真正跨块优化，需先建 SSA 或到达定义。

## 瓶颈聚类与并行分工

四条工作流互相**文件隔离**，可并行无冲突。每条流一个 `feat/*` 分支 + 独立 worktree。

---

### 流 A：函数内联强化　`feat/perf-inline`
**目标基准**：bench_call_intensive(0.601)、helper_loop(0.667)、fib(0.512)、bench_matrix(0.423)、p09(0.765)
**主要文件**：`src/opt/Passes.cpp`（inlineFunctions / inlinable / inlineSweep）
**现状**：内联门槛保守——仅单基本块、≤24 指令、非递归、非 main。
**改进方向**：
1. 放宽指令数门槛（如 ≤60），并允许**多基本块**小函数内联（需做 CFG 拼接：把 callee 的
   块重映射 id 后插入 caller，调用点拆块，return 改跳转到续延块）。
2. 对**叶子小函数**（无调用、单 return）优先内联，收益最大（helper_loop 的 sq、bench_matrix 的 elem）。
3. 多轮内联收敛后交给现有 DCE/copyProp 清理。
**风险**：多块内联涉及 CFG 重建，务必保持 vreg 重命名一致、term 正确；用 p09/fib 验证退出码不变。

---

### 流 B：尾递归消除　`feat/perf-tailcall`
**目标基准**：p06_tail_recursion(0.454)、间接利好 fib/p09 递归类
**主要文件**：`src/opt/Passes.cpp`（新增 pass）
**现状**：尾递归 `return f(...)` 按普通调用展开，每层一个栈帧 + call/ret。
**改进方向**：
1. 识别 `return self_call(args)` 形式的**自尾递归**（callee == 当前函数且在 return 位置）。
2. 改写为：把实参赋值回形参 vreg → 跳转到函数入口块（需在入口前插一个 landing 块）。
3. 注意非 SSA：形参 vreg 重赋值要用临时寄存器避免参数互相覆盖（如 `f(b,a)` 交换）。
**风险**：只处理直接自递归；互递归 ToyC 不支持。用 p06 验证（500 层深度退出码不变、指令数大降）。

---

### 流 C：跨块/循环优化　`feat/perf-loopopt`
**目标基准**：p04_common_subexpr(0.500)、p07_loop(0.769)、p02_dead_code(0.667)、nested(0.857)
**主要文件**：`src/opt/Passes.cpp`（licm / 新增到达定义分析 / 跨块 CSE）
**现状**：CSE、copyProp 仅块内；LICM 已修复但只提单层不变量。
**改进方向**：
1. **加强 LICM**：当前要求操作数循环内从不定义。可扩展到"操作数的定义本身已被提到
   preheader"的链式提升（多层不变量）；并支持把循环内**不变的乘除**也提出。
2. **循环内 CSE**：循环体是独立块，块内 CSE 已生效，但跨"体块↔条件块"的公共子表达式
   未消除。需谨慎（非 SSA），建议先做保守的"循环不变子表达式提升合并"。
3. 提升后必须复跑 LICM 正确性用例（licm_invariant / licm_global_barrier）。
**风险**：跨块最容易踩非 SSA 坑。每步都跑全量 -opt + opt_compare。

---

### 流 D：Codegen 与强度削减　`feat/perf-codegen`
**目标基准**：arith(0.889)、divmod(0.765)、bench_reg_pressure(0.800)、p11(0.762)
**主要文件**：`src/codegen/OptCodeGen.cpp`、`src/opt/RegAlloc.cpp`
**现状**：乘常量已做移位化简；除/模常量已做 magic-number；寄存器分配为图着色+合并。
**改进方向**：
1. **算术序列**：审视 arith 的循环体，看是否有可合并的 addi/移位序列（窥孔层）。
2. **全局常量传播**（p11）：全局变量若在 main 设定后只读，可在 codegen 把 LoadGlobal
   折成立即数——但需全局只写一次的静态判定（保守：仅当全局在所有函数中无 StoreGlobal
   或仅 main 顶层赋常量）。也可放 opt 层。
3. **寄存器压力**（bench_reg_pressure）：检查溢出选择启发式，减少 spill load/store。
**风险**：codegen 改动影响所有用例，全量回归必跑。

---

## 协作流程（GitFlow-lite）

```
develop (基线，59/59，baseline 已记录)
  ├── feat/perf-inline     (流A)
  ├── feat/perf-tailcall   (流B)
  ├── feat/perf-loopopt    (流C)
  └── feat/perf-codegen    (流D)
```

每条流完成后：
1. 在自己 worktree 跑 `perf-tester` 代理：构建 → 功能 59/59 → opt_compare 0 BUG → bench_icount 对比 baseline。
2. ratio 有提升且无退化 → `--no-ff` 合并回 develop。
3. 合并顺序建议 D→A→B→C（codegen 先稳，再上内联，最后跨块），每次合并后复测全量。

## worktree 创建命令

```sh
cd /d/Toc/branches/develop
for s in inline tailcall loopopt codegen; do
  git branch feat/perf-$s develop
  git worktree add ../perf-$s feat/perf-$s
done
```
