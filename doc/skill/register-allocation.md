# 图着色寄存器分配：关键点与一个隐蔽 bug

## 流程骨架

1. **活跃性分析**：CFG 上后向数据流求 live-in/live-out。
   `live_out[b] = ∪ live_in[succ]`，`live_in[b] = use[b] ∪ (live_out[b] − def[b])`，迭代至不动点。
2. **构造冲突图**：反向扫每个块，维护"当前活跃集"。在定值点，被定值寄存器与当前活跃的
   其它寄存器两两冲突。
3. **着色**：给每个节点选一个不与邻居同色的物理寄存器；选不出就溢出。

## 让"传送可合并"——不冲突 + 偏置着色

- 对 `mv d, s`：**不要**让 `d` 与 `s` 冲突（即在 `d` 的定值点把 `s` 排除在外）。
- 着色时带**偏置**：给 `d` 选色时优先选 `s` 的颜色（若可用），从而 `mv` 退化为同寄存器、
  被消除。这是"穷人的合并"，简单且有效。

## 跨调用的值 → 必须用被调用者保存寄存器

调用会破坏调用者保存寄存器。**凡跨越调用而活跃的虚拟寄存器，限制其只能着色到
`s0–s11`**（否则被调用破坏），不足再溢出。实现上：扫到 `Call` 指令时，把"此刻仍活跃
（即调用之后还要用）"的虚拟寄存器标记为 callCrossing，着色时它们的候选集合只含被调用者
保存寄存器。

偏好顺序：不跨调用的值优先用调用者保存寄存器（免保存开销），跨调用的值才用被调用者保存
寄存器——这样能少存少取。

## 一个隐蔽 bug：形参之间漏建冲突

**症状**：一个九参数函数 `f(a,…,i)`，函数体里 `a+b+…+i` 同时用到全部形参，结果它们被
分配到了**同一个寄存器**，函数返回完全错误的值；递归函数也因形参互相覆盖而无限递归。

**根因**：冲突是在"定值点"建立的，而**形参从不是任何 IR 指令的定值目标**（它们在函数
入口被隐式定义）。于是基于定值的规则**永远不会在形参之间建边**，分配器以为它们能共享
寄存器。

**修复**：显式地让所有形参两两冲突（它们在入口同时活跃），并让形参与"入口处活跃的其它
值"冲突。一行补丁，但若没有"多参数 + 差分测试"很难发现——这正是模糊测试的价值。

## 溢出别忘了

即使预期很少溢出，也要把溢出路径写对并测到（构造 >可用寄存器数 个同时活跃的变量强制
触发）。预留固定的 scratch 寄存器给"溢出操作数的临时装载"，且要能容纳一条二元指令同时
有两个溢出操作数的情形。
