# 实现 C 子集编译器的整体经验

## 结论先行

1. **从能跑通的最小后端开始，再谈优化**。先用"每个虚拟寄存器一个栈槽"的朴素后端把
   端到端打通、测试到 100% 通过，再把优化后端作为可切换的第二条路径——朴素后端从此
   成为差分测试的黄金参照，几乎零成本地为后续每一次优化兜底。
2. **正确性优先于一切**。评测里功能分通常占大头，一个为了性能引入的回归可能赔掉远超
   收益的分数。每加一个优化，立刻用差分测试回归。
3. **"文法"与"语义"冲突时，以语义/参照实现为准**。

## 几个具体踩坑

### 优先级：文法同级 ≠ C 语义同级
ToyC 文法把 `< > <= >= == !=` 列在同一层，但要求"运行结果与 C 一致"，而 C 里
`==/!=` 优先级低于关系运算。`a == b < c` 两种解释结果不同。**当题面同时给出形式文法
和"与某参照语言一致"的语义约束时，二者可能矛盾；应以语义/参照为准**，否则会在混合
比较的用例上失分。解决：把相等运算单独下沉一层。

### 一元负号与 `INT_MIN`
若文法的整数字面量带可选负号，但负号又是一元运算符，按"词法只认无符号数字串、负号交给
语法"处理最干净。数值用 64 位解析再截成 32 位补码，`-2147483648` 即"一元负
`2147483648`"，靠补码 `-INT_MIN==INT_MIN` 自然得到正确值。

### "先消费再交给子过程消费"——token 二次消费
递归下降里一个常见 bug：在分发处 `advance()` 吃掉了 `int`，被调用的 `parseGlobalVar`
里又 `expect(int)`，导致二次消费。**约定好"谁来消费哪个 token"并保持一致**；用多看
一两个 token（`peek(2)`）来分发，避免提前消费。

### 不要误拒合法程序
当题目保证"输入全部合法、无 UB"时，语义分析应以"解析正确"为目标，**对难以零误报判定
的性质（如全路径 return）宁可不强制拒绝**，把兜底放到代码生成（控制流落到函数尾部时补
一条默认返回）。分析器自身的保守性一旦误杀合法程序，就是实打实的失分。

## 架构取舍

- **手写词法/语法 vs 生成器**：对这种小而规整的文法，手写递归下降更可控、无外部依赖、
  报错可读，且便于在语义阶段做即时折叠。
- **要不要上 IR**：上。即便是简单语言，一层三地址 IR + CFG 能让寄存器分配、循环优化、
  强度削减各自独立、可测，远比"AST 直接吐汇编"可维护。
- **变量即虚拟寄存器**：没有取地址的语言里，局部变量天然不必进内存，全部建模为虚拟
  寄存器，把"放寄存器还是溢出"完全交给分配器——这是后端质量的关键前提。
