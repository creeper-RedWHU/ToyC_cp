## 一、测试架构

### 1.1 差分测试原理

ToyC 是 C 语言的严格子集，同一源文件可以用宿主 C 编译器直接编译运行。测试框架利用这一特性实现"差分测试"：

```
ToyC 源文件
    ├─▶ 宿主 gcc (x86)   ──▶ 原生执行 ──▶ 退出码（期望值）
    └─▶ ToyC 编译器       ──▶ RV32 汇编 ──▶ riscv-gcc 链接 ──▶ Spike 模拟 ──▶ 退出码（实际值）
                                                                    ↑
                                                               比对低 8 位
```

当两个退出码的低 8 位相同时，测试通过。

### 1.2 测试工具

| 工具 | 说明 |
|------|------|
| `tests/run_tests.sh` | 主功能测试脚本，对 `tests/cases/*.tc` 逐一运行差分测试 |
| `tests/run_opt_compare.sh` | 优化正确性对比脚本，验证 `-opt` 和无 `-opt` 结果一致 |
| `tests/bench.sh` | 性能基准脚本，对比 ToyC compiler (-opt) 与 gcc -O2 的运行时间 |
| `tests/fuzz.py` | 差分模糊测试，随机生成合法 ToyC 程序并验证 |

### 1.3 脚本使用方法

```bash
# 构建编译器
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build -j

# 功能测试（全部用例）
tests/run_tests.sh

# 功能测试（带优化）
tests/run_tests.sh -opt

# 只测某类用例（如内联相关）
tests/run_tests.sh -opt "inline_*.tc"

# 优化正确性对比（验证 -opt 不改变结果）
tests/run_opt_compare.sh

# 只对比优化相关用例
tests/run_opt_compare.sh "inline_*.tc"

# 性能基准测试
tests/bench.sh

# 差分模糊测试（300 个随机程序）
tests/fuzz.py 300

# 模糊测试（含递归，带优化）
tests/fuzz.py 100 --opt
```

### 1.4 环境依赖

| 依赖 | 用途 | 备注 |
|------|------|------|
| `riscv64-unknown-elf-gcc` | 汇编 + 链接 | `-march=rv32im -mabi=ilp32` |
| `spike` | RV32IM 模拟执行 | `--isa=rv32im` |
| `cc` / `gcc` | 差分期望值计算 | 宿主编译器 |

### 1.5 用例注释约定

用例文件可在首行用 `// expect: N` 固定期望退出码，避免依赖宿主编译器：

```c
// expect: 42
int main() { return 42; }
```

## 三、优化正确性验证（run_opt_compare.sh）

### 3.1 用途

`tests/run_opt_compare.sh` 专门验证优化不改变程序语义。对每个功能测试用例：

1. 用**无 `-opt`** 编译，在 Spike 上运行，记录退出码 `plain`
2. 用**`-opt`** 编译，在 Spike 上运行，记录退出码 `opt`
3. 若 `plain ≠ opt`，报告 **OPT_BUG**

### 3.2 针对优化 pass 的专项用例

| 优化 Pass | 专项用例 | 验证方式 |
|-----------|---------|---------|
| 函数内联 | `inline_boundary`, `inline_no_recurse`, `inline_variable_shadow` | `run_opt_compare.sh inline_*.tc` |
| LICM | `licm_invariant`, `licm_global_barrier` | `run_opt_compare.sh licm_*.tc` |
| 图着色寄存器分配 | `spill_correctness_large`, `spill_stress`, `stack_mixed_params` | `run_opt_compare.sh *spill*.tc *stack*.tc` |
| Return Coalescing | `return_coalescing` | `run_opt_compare.sh return_coalescing.tc` |
| Magic-div | `magic_div_edge`, `magic_exhaustive`, `divmod_neg` | `run_opt_compare.sh *magic*.tc *divmod*.tc` |
| Loop Rotation | `triple_nested_loop`, `nested_loops`, `while_true_break` | `run_opt_compare.sh *loop*.tc *while*.tc` |

### 3.3 建议 CI 流程

```bash
# 1. 功能全套（无优化）
tests/run_tests.sh

# 2. 功能全套（带优化）
tests/run_tests.sh -opt

# 3. 优化一致性对比（所有用例，关键门禁）
tests/run_opt_compare.sh

# 4. 差分模糊（快速，100 个随机程序）
tests/fuzz.py 100 --opt
```

## 四、性能基准测试（9 个）

### 4.1 评分方式

$$\text{性能得分} = \frac{\sum \min\!\left(1,\ \frac{t_{\text{gcc}-O2}}{t_{\text{ours}}}\right)}{N} \times 100$$

每个用例得分 = `min(1.0, gcc-O2时间 / 我们的时间)`，满分封顶。

### 4.2 原有基准（5 个）

| # | 文件名 | 测试重点 | 优化收益 |
|---|--------|---------|---------|
| 1 | `bench/arith.tc` | 2000万次乘除混合 | 常量强度削减（`*3`, `/2` → 移位） |
| 2 | `bench/divmod.tc` | 800万次变量除法 + 常量 `%7` | magic-div 替换 |
| 3 | `bench/fib.tc` | `fib(34)` 深度递归 | 寄存器分配（减少 spill）、内联（无收益，递归） |
| 4 | `bench/helper_loop.tc` | 热循环调用 `sq()`, `lin()` | 函数内联（消除调用开销） |
| 5 | `bench/nested.tc` | 4000×4000 嵌套循环 + `%7` | LICM、loop rotation、magic-div |

### 4.3 新增基准（4 个）

| # | 文件名 | 迭代规模 | 测试重点 | 优化收益分析 |
|---|--------|---------|---------|------------|
| 6 | `bench/bench_sort.tc` | 50万次随机比较交换 | 分支预测友好性、LICM | loop rotation 减少冗余跳转 |
| 7 | `bench/bench_matrix.tc` | 16×16 矩阵乘法 | 三层嵌套循环 + 函数调用 | LICM 提升 `elem_a(i,k)` 的不变计算；内联 `elem_a/b` |
| 8 | `bench/bench_call_intensive.tc` | 300万次迭代 | 函数调用密集（4 个小函数/迭代） | 内联 `add/mul2/sub1/clamp100` 消除调用开销 |
| 9 | `bench/bench_reg_pressure.tc` | 100万次迭代 | 高寄存器压力（20+ 活跃变量/块） | 图着色分配质量，spill 减少 |
