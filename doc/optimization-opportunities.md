## 优先级 P0 — 高收益，低难度

### 1. 冗余移动消除（Copy Propagation） ✅ 已完成

**实现**：按基本块前向扫描，维护 `copyOf: vreg → 源 vreg` 映射表。遇到 `mv d, s` 时记录 `d` 是 `s` 的副本；遵循映射链（`r3→r2, r2→r1` → `r3→r1`）；当源 vreg 被非 mv 指令重新定义时，所有引用该源的副本立即失效。替换所有随后的寄存器使用为最终源 vreg。

**影响力分析**：与寄存器分配器已有的 move biasing 互补。IR 层的传播可级联消除多层 mv，且减少虚拟寄存器数量，降低后续寄存器分配的压力。DCE 自动清理传播后不再被使用的 mv 指令。

**添加流程**：`algebraicSimplify → copyPropagation → deadCodeElim → (opt) hoistLoopConstants → deadCodeElim`

---

### 2. 代数化简（Algebraic Simplification） ✅ 已完成

**实现**：按基本块扫描，匹配以下恒等模式并原地改写指令 op 与操作数：

| 模式 | 化简为 | 说明 |
|------|--------|------|
| `x + 0` / `0 + x` | `x` (Mv) | 加法单位元 |
| `x - 0` | `x` (Mv) | |
| `0 - x` | `-x` (Neg) | |
| `x * 0` | `0` | |
| `x * 1` / `1 * x` | `x` (Mv) | |
| `x * -1` / `-1 * x` | `-x` (Neg) | |
| `x / 1` | `x` (Mv) | |
| `x / -1` | `-x` (Neg) | |
| `x % 1` / `x % -1` | `0` | |
| `x - x` | `0` | 相同 vreg 时 |
| `Neg(Neg(x))` | `x` (Mv) | 通过前向 def 链检测 |
| `Lt/Gt/Ne(x, x)` | `0` | 相同 vreg 时 |
| `Ge/Le/Eq(x, x)` | `1` | 相同 vreg 时 |

**注**：`x - x` 在相同 vreg 时无条件成立（一个寄存器在同一条指令中两次读取得到相同值）。`x/x` 和 `x%x` 因除零 UB 前提而不做化简。

### 3. 本地公共子表达式消除（Local CSE）

**现状**：完全没有 CSE。相同的表达式会重复计算。

**现状**：完全没有 CSE。相同的表达式会重复计算。

**例子**：
```
r1 = x + y
// ... 中间没有修改 x 或 y ...
r2 = x + y    ← 完全相同，可复用 r1
```

**方法**：在每个基本块内做 value numbering — 维护一个 `{op, a, b} → dst` 的映射表，对每个二元/一元指令检查是否已存在相同计算
**位置**：IR 层，[opt/Passes.cpp](src/opt/Passes.cpp)，作为单独 pass
**效果**：在含重复子表达式的代码中效果显著

### 4. 空块消除 + 块合并

**现状**：IR 生成和优化后可能留下只含无条件跳转的空块。

```
.L1:          ← 空块，只跳到 L2
  j .L2
.L2:
  add ...
```

**消除**：将所有跳转到 L1 的地方直接改为跳转 L2，删除 L1 块，更新所有后继/前驱列表。
**块合并**：若块 A 的**唯一后继**是块 B，且块 B 的**唯一前驱**是块 A，则可合并两者，消除 A 末尾的 `j` 指令。
**效果**：功能测试和性能测试都能受益，减少分支指令数

## 优先级 P1 — 高收益，中等难度

### 5. 稀疏条件常量传播（SCCP）

**现状**：常量折叠仅在单个表达式内（Sema 阶段），不跨基本块传播。

**例子**：
```c
int x = 5;
if (x > 3) {       // 折叠为 true，整个 if 可消除
    return x + 1;  // 折叠为 6
} else {
    return 0;      // 死代码
}
```

**方法**：使用 SSA-like 的 lattice 传播（TOP→常量→BOTTOM），结合分支折叠，传播常量到整个函数。ToyC 的 IR 虽非 SSA 形式，但可在每个基本块入口维护已知常量集做前向传播。
**效果**：消除不可达分支和常量计算，对功能测试也有显著帮助

---

### 6. 分支/跳转目标折叠（Branch-to-Branch Resolution）

**现状**：如果块 A 跳转到块 B，块 B 又跳转到块 C，A 应该直接跳转到 C。

```
.L1: j .L2
.L2: j .L3    ← 中间块
.L3: ...
```

**方法**：对每个块的 term 指令，解析其跳转目标是否为另一个仅含无条件跳转的块，如是则直达最终目标
**位置**：IR 层，可合并进空块消除 pass

## 优先级 P2 — 特定场景优化，中等难度

### 7. 尾调用优化（Tail Call Optimization）

**现状**：`return g(args);` 生成 `call g` → `mv a0, ret_reg` → `ret`，多了一对 call/ret。

**优化**：恢复 callee-saved 寄存器 → 恢复 ra → 调整 sp → `j g`（复用当前栈帧）。
**条件**：被调函数签名兼容，且调用是函数末尾的最后操作（没有后续指令再用返回值做计算）。
**位置**：[codegen/OptCodeGen.cpp](src/codegen/OptCodeGen.cpp) — `emitCall` 和 `emitTerm` 中检测尾调用模式
**效果**：减少调用深度，消除 call/ret 对的开销

---

### 8. 小循环展开（Loop Unrolling）

**现状**：while 循环每次迭代执行条件判断和分支。

**适用场景**：迭代次数较小且可静态确定（或被调用前已知范围）、循环体较小。
**例子**：
```c
int i = 0; while (i < 4) { s = s + i; i = i + 1; }
// 展开为:
s = s + 0; s = s + 1; s = s + 2; s = s + 3;
```

**注意**：ToyC 无数组，但累加器模式的循环常见。需要处理 break/continue。
**位置**：IR 层，也可在 AST 到 IR 生成时做

## 优先级 P3 — 中高收益，较高实现难度

### 9. 全局变量寄存器提升（Global Variable Promotion）

**现状**：每次全局变量读写发射 `lui + lw` / `lui + sw` 序列（2 条/次）。

**优化**：若一个函数内多次读写同一全局变量，可在入口处一次加载到 vreg，出口处存回。需要确保其间没有调用的函数会修改该全局变量。
**位置**：IR 层分析 pass → 替换 LoadGlobal/StoreGlobal 为 Mv
**效果**：对多全局变量访问的代码效果明显（如 globals.tc 测试用例）

### 10. 寄存器分配改进

**现状**：贪心图着色（[opt/RegAlloc.cpp:132-167](src/opt/RegAlloc.cpp#L132-L167)），spill 策略简单。

**可改进的方面**：
- **Spill 按使用频率排序**：当前按固定顺序（`adj[a].size() > adj[b].size()`），可改为按实际使用次数加权
- **Live range splitting**：不同区域用不同寄存器，避免全局寄存器压力扩散
- **复用 spill slot**：不同时 live 的 vreg 共享同一 spill slot，减少栈帧大小
- **a0-a7 临时分配**：对短生命周期的值，也可使用 a0-a7（当前完全排除）

**位置**：[opt/RegAlloc.cpp](src/opt/RegAlloc.cpp)
**效果**：减少 spill/reload 次数，间接提升所有场景性能

---

### 11. 块布局优化（Block Placement）

**现状**：基本块按 IR 生成顺序排列，fall-through 只依赖相邻关系。

**优化**：分析分支概率（或静态启发式：循环头比循环尾热、then 比 else 热），重新排列基本块，让最热路径 fall-through，冷块移出热路径减少 I-cache 压力。
**位置**：[codegen/OptCodeGen.cpp:196-208](src/codegen/OptCodeGen.cpp#L196-L208) — `emitFunc` 中的 block ordering

---

### 12. 改进内联策略

**现状**：只内联单基本块、≤24 条指令、非递归、非 main 的函数（[opt/Passes.cpp:22-25](src/opt/Passes.cpp#L22-L25)）。

**可改进**：
- 多基本块但总指令数很小的函数
- 基于调用次数的成本模型（被多次调用的函数应优先内联）
- 内联后对调用点进行常数传播（Callsite specialization）
